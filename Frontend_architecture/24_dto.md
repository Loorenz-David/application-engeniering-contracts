# 24 — DTO Contract

## Definition

A DTO (Data Transfer Object) is a plain, immutable data shape that crosses a system boundary. In the frontend, the boundary is the HTTP layer. DTOs carry data in — they never carry behavior. They are never mutated and never extended with computed properties.

This contract defines the four DTO categories, their naming rules, and the transformation pipeline that converts raw network data into what the UI consumes.

---

## The four categories

| Category | Direction | Named | Lives in |
|---|---|---|---|
| **Response DTO** | Backend → Frontend | `<Entity>` | `feature/types.ts` |
| **Request DTO** | Frontend → Backend | `<Action><Entity>Input` | `feature/types.ts` |
| **Query Params DTO** | Frontend → Backend (URL) | `<List><Entity>Params` | `feature/types.ts` |
| **View Model** | Internal only | `<Entity>ViewModel` | `feature/types.ts` |

A View Model is **not** a DTO — it does not cross a system boundary. It is a derived type, computed from a Response DTO inside the controller, for display purposes only. It is listed here because it completes the pipeline.

---

## The transformation pipeline

```
HTTP response (raw JSON)
        │
        │  Zod parse in API function
        ▼
Response DTO  (e.g. Invoice)
        │
        │  toXxxViewModel() in controller
        ▼
View Model  (e.g. InvoiceViewModel)
        │
        │  injected via context
        ▼
Component  (renders)
```

And for the write path:

```
Component fires callback
        │
        │  action receives typed input
        ▼
Request DTO  (e.g. CreateInvoiceInput)  ← validated by Zod in form hook
        │
        │  API function sends as JSON body
        ▼
HTTP request (raw JSON)
        │
        ▼
Backend
```

## Category 1 — Response DTO

Represents the exact shape the backend returns for one entity. Defined with a Zod schema; the TypeScript type is always `z.infer` of that schema.

```ts
// features/invoices/types.ts

export const InvoiceSchema = z.object({
  // Public client-facing ID. This is not the backend database primary key.
  id:           z.string().uuid().transform((v) => v as InvoiceId),
  number:       z.string(),
  status:       z.enum(['draft', 'sent', 'paid', 'overdue']),
  amount_cents: z.number().int().nonnegative(),
  due_date:     z.string().datetime({ offset: true }),
  customer_id:  z.string().uuid().transform((v) => v as CustomerId),
  created_at:   z.string().datetime({ offset: true }),
});

export type Invoice = z.infer<typeof InvoiceSchema>;
```

**Rules:**
- Field names mirror the backend serializer exactly — snake_case to match the JSON
- Monetary values are stored in the smallest unit (`amount_cents`, not `amount`)
- Dates are ISO strings — never `Date` objects (they cannot be safely serialized)
- IDs are public client-facing IDs and are cast to branded types at the schema boundary (`.transform((v) => v as InvoiceId)`)
- The schema lives in `types.ts`; the Zod parsing happens in the API function (`api/fetch-invoice.ts`), never in the component
- Response DTOs never expose backend database primary keys

## Category 2 — Request DTO

Represents the payload sent to the backend for a write operation. One schema per mutation endpoint.

```ts
// features/invoices/types.ts

export const CreateInvoiceInputSchema = z.object({
  // client_id is generated by the frontend (crypto.randomUUID()) and sent to the backend.
  // The backend stores it as the entity's public client-facing identifier and returns
  // that same value as the response DTO's public id.
  client_id:   z.string().uuid(),
  customer_id: z.string().uuid({ message: 'Select a customer.' }),
  due_date:    z.string().datetime({ offset: true, message: 'Enter a valid due date.' }),
  notes:       z.string().max(500).optional(),
  line_items:  z.array(
    z.object({
      description:      z.string().min(1, 'Description is required.'),
      quantity:         z.number().int().positive('Quantity must be at least 1.'),
      unit_price_cents: z.number().int().nonnegative('Price cannot be negative.'),
    })
  ).min(1, 'Add at least one line item.'),
});

export type CreateInvoiceInput = z.infer<typeof CreateInvoiceInputSchema>;

export const UpdateInvoiceInputSchema = z.object({
  id:         z.string().uuid().transform((v) => v as InvoiceId),
  notes:      z.string().max(500).optional(),
  line_items: z.array(z.object({
    description:      z.string().min(1),
    quantity:         z.number().int().positive(),
    unit_price_cents: z.number().int().nonnegative(),
  })).optional(),
});

export type UpdateInvoiceInput = z.infer<typeof UpdateInvoiceInputSchema>;
```

**Rules:**
- One schema per operation — `CreateXxxInput` is never reused for `UpdateXxxInput`
- Every first-party Create Request DTO includes `client_id: z.string().uuid()` — see the Client-generated IDs section below
- Every Update Request DTO includes `id` so the action hook can apply the optimistic update before the request resolves
- Validation messages are user-facing strings — write them as a user would read them
- The schema is the single source of truth for form validation — `zodResolver` points to it
- Field names match what the backend endpoint expects in the request body
- Never send a View Model as a request body — it contains display fields the backend did not define

---

## Category 3 — Query Params DTO

Represents URL query string parameters for list or filtered endpoints. These are always optional; unset params are omitted from the URL.

```ts
// features/invoices/types.ts

export type ListInvoicesParams = {
  page?:        number;
  per_page?:    number;
  status?:      Invoice['status'];
  search?:      string;
  customer_id?: CustomerId;
};
```

**Rules:**
- All fields are optional — the endpoint must work with any subset
- Types match what the API accepts: `number` for pagination, `string` for filters
- No Zod schema is required unless the params come from user-editable input that needs validation (e.g. a URL parsed on load). In that case, validate with Zod before use.
- Undefined values are omitted from the URL (the API client strips them — see `04_api_client.md`)

## Category 4 — View Model

A View Model is a Response DTO extended with computed, display-ready fields. It is not validated with Zod because it is computed internally, not received over the network.

```ts
// features/invoices/types.ts

export type InvoiceViewModel = Invoice & {
  amount_formatted:  string;          // "$1,234.56"
  status_label:      string;          // "Overdue"
  status_variant:    'success' | 'danger' | 'warning' | 'default';
  is_overdue:        boolean;
  due_in_days:       number | null;   // null when paid
};

export function toInvoiceViewModel(invoice: Invoice): InvoiceViewModel {
  const now     = new Date();
  const dueDate = new Date(invoice.due_date);
  const days    = Math.ceil((dueDate.getTime() - now.getTime()) / 86_400_000);

  return {
    ...invoice,
    amount_formatted: new Intl.NumberFormat('en-US', {
      style: 'currency', currency: 'USD',
    }).format(invoice.amount_cents / 100),
    status_label:   { draft: 'Draft', sent: 'Sent', paid: 'Paid', overdue: 'Overdue' }[invoice.status],
    status_variant: invoice.status === 'paid' ? 'success'
                  : invoice.status === 'overdue' ? 'danger'
                  : 'default',
    is_overdue:  invoice.status === 'overdue',
    due_in_days: invoice.status === 'paid' ? null : days,
  };
}
```

**Rules:**
- The transformer (`toXxxViewModel`) is a pure function — no async, no side effects
- It lives in `types.ts` alongside the DTO it derives from
- It is called inside the controller, not in a component or API function
- The View Model extends the Response DTO via intersection (`Invoice &`) — it never drops DTO fields
- Never add backend-domain fields to the View Model — only display-derived fields belong here

## Where DTOs are parsed

Parsing happens once, in the API function, at the HTTP boundary. Nothing else calls the Zod schema:

```ts
// features/invoices/api/fetch-invoice.ts
import { z } from 'zod';
import { apiClient } from '@/lib/api-client';
import { InvoiceSchema } from '@/features/invoices/types';

const GetInvoiceResponse = z.object({ invoice: InvoiceSchema });

export async function fetchInvoice(id: InvoiceId) {
  const { invoice } = await apiClient.get(
    `/api/v1/invoices/${id}`,
    GetInvoiceResponse,
  );
  return invoice;  // type: Invoice (Response DTO)
}
```

The query hook calls the API function and gets back a typed Response DTO. The controller calls the query hook and calls `toInvoiceViewModel` on the result. The component reads from context and receives a View Model.

---

## DTO alignment with backend

Response DTOs are contracts with the backend. When the backend changes a field name, type, or adds a required field, the frontend DTO schema breaks loudly at the Zod parse step — not silently at render time. This is the point.

Treat the Zod schema as documentation of the backend contract. A field that the backend returns but the schema does not declare is silently dropped by Zod's `strip` mode (the default). Declare every field the app needs.

---

## Client-generated IDs (`client_id`)

Every first-party Create Request DTO includes a `client_id` field — a UUID generated by the frontend using `crypto.randomUUID()` before the request fires. The backend stores this in the entity's `client_id` column, which is the external-facing public identifier. The backend's internal database `id` remains private.

This rule applies to entities owned by the application database. It does not apply to resources whose IDs must be assigned by an external provider, payment processor, identity provider, or legacy backend contract. Those exceptions must be documented in the feature's `types.ts` near the request schema.

```ts
// features/invoices/types.ts
export function buildCreateInvoiceInput(
  formData: Omit<CreateInvoiceInput, 'client_id'>,
): CreateInvoiceInput {
  return CreateInvoiceInputSchema.parse({
    ...formData,
    client_id: crypto.randomUUID(),
  });
}
```

```ts
// The controller or flow calls the DTO builder before firing the action.
const input = buildCreateInvoiceInput(formData);
const id    = input.client_id as InvoiceId;

createAction.createInvoice(input);   // fire — do not await
navigate(ROUTES.invoiceDetail(id));  // navigate immediately with the known ID
```

The schema parse happens inside the DTO builder, which is the construction boundary for generated fields. Controllers and components do not call arbitrary Zod schemas directly.

**Why this enables optimistic updates:**

Without `client_id`, the frontend must wait for the server response to learn the entity's public ID. The optimistic cache entry gets a temp ID that must be swapped for the real one after the response — fragile and complex.

With `client_id`, the frontend knows the ID before the request fires:

```
t=0ms  → ID generated: "abc-123"
t=0ms  → Optimistic cache entry added with id "abc-123"
t=0ms  → User navigated to /invoices/abc-123 — page renders from cache
t=0ms  → POST /invoices fires with { client_id: "abc-123", ... }
t=150ms → Server responds with { invoice: { id: "abc-123", ... } }
t=150ms → Detail cache seeded — no ID swap, no extra fetch
```

**Naming convention:**

`client_id` in a Request DTO means "this entity's public client-facing ID, proposed by the client application." It does not refer to a customer or client entity. Use `customer_id`, `account_id`, or the appropriate domain term for entity relationship foreign keys.

**Server contract:**

The backend must be built to accept `client_id` and persist it as the entity's external-facing UUID identifier. It is unique and indexed, but it is not the internal database primary key. Duplicate `client_id` values (retried requests) are handled by the backend's idempotency logic — the backend returns the existing entity rather than creating a duplicate.

## What DTOs must NOT do

- **Response DTOs must never contain computed or display-only fields.** Those belong in the View Model. The DTO reflects only what the backend sent.
- **View Models must never be sent as request bodies.** Transform back to a Request DTO first.
- **DTOs must never be mutated.** They are immutable once parsed. Derive a new object instead.
- **Parsing must never happen inside a render component or business controller.** Parse at the boundary: API function, form resolver, DTO builder for generated fields, URL-param parser, storage helper, env parser, or browser event handler.
- **Dates must never be stored as `Date` objects.** Store as ISO strings in DTOs; format in the View Model transformer or in a utility function.
- **Monetary values must never be stored as floats in DTOs.** Use integer cents. Convert to formatted strings only in the View Model.
- **A Request DTO schema must never be reused as a Response DTO schema.** They have different validation semantics — inputs are strict; outputs are permissive on optional fields.
