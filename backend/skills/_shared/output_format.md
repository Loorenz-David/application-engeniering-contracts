# Backend Skill Output Format

Every backend skill response should use this structure before implementation.

## 1) Contracts loaded

- `<path>`: `<why needed>`

## 2) Local extensions loaded

- `<canonical_local.md>`: `<app-specific delta>`

## 3) Plan

1. `<step>`
2. `<step>`

## 3a) Clarifications needed (only when ambiguous)

- `<question>`: `<why this blocks safe implementation>`

If this section is non-empty, implementation planning must stay draft-only until
the user answers.

## 4) Changes made

- `<file>`: `<what changed>`

## 5) Validation

- `<command>`: `<result>`

## 6) Exclusions

- `<contract or path not loaded>`: `<reason>`
